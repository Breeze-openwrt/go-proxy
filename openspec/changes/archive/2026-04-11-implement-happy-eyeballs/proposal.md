# Proposal: Implement Happy Eyeballs v2 (Concurrent Dialing)

## 1. 问题定位 (Problem)
目前的 Dial 逻辑是线性的：解析 DNS -> 拿到第一个 IP -> 发起 TCP 握手。
- **缺点**: 如果该 IP 正在抖动，或者 IPv6 路由不可达，程序会卡死在 Timeout 周期内。
- **现状**: 即使有连接池，在池子空了或者异步补货时，这种线性延迟依然会渗透给终端用户。

## 2. 改进规格 (Specs)
- **多路并发**: 针对一个目标地址，解析出所有可用 IP (IPv4/IPv6)。
- **赛马竞争 (Racing)**: 顺序开启多个 Dial 任务，每个任务之间有 250ms 的延迟（防止 SYN Flood），取最快完成握手的连接。
- **极致收敛**: 获胜者产生后，毫秒内关闭并销毁所有其他正在进行的握手。

## 3. 交付价值
- 彻底消灭因单点故障或线路抖动导致的“首字节加载变慢”。
- 自适应最优路径（自动在 IPv4 和 IPv6 之间选择最快路径）。
