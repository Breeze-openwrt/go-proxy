# Proposal: Implement Elastic Backend Connection Pool

## 1. 问题定位 (Problem)
当前的 `BackendPool` 实现存在以下缺陷：
- **无复用**: `Put` 操作直接 `conn.Close()`，导致每个请求都必须经历 TCP 十五字节的三次握手。
- **僵尸连接**: 预热产生的连接在池中可能因为后端超时而失效，导致 `Get` 到死连接。
- **由于缺乏长连接利用率**，在高并发下 QPS 受限于握手开销。

## 2. 改进规格 (Specs)
- **真复用**: 优化 `Put` 逻辑，在池容量允许且连接健康的情况下将其回收。
- **空闲超时**: 引入 `MaxIdleTimeout` (默认 60s)，自动剔除池中过久的连接。
- **自愈能力**: `Get` 操作增加连通性嗅探，发现失效连接自动重连。
- **弹性伸縮**: 允许在不增加延迟的情况下动态处理突发流量。

## 3. 交付价值
- 连接复用大幅降低后端响应延迟。
- 减少系统的 FD (文件描述符) 抖动。
